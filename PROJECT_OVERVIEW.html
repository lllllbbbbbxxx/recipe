一、项目概述

这是一个帮助用户决定今天吃什么的微信小程序。核心流程：录入会做的菜 → 记录冰箱库存 → 根据库存智能推荐/随机抽取今日菜单 → 汇总购物清单。数据同时存本地缓存（wx.getStorageSync）和微信云数据库，支持多设备同步与备份/恢复。

五、核心逻辑说明

匹配算法（match-utils.js）

把菜谱食材分为 4 级：main（主料）→ required（必要）→ optional（配料）→ seasoning（调料）。与冰箱食材做模糊匹配（包含关系）。

  • canCook = 全部齐
  • canDraw = 主料+必要料齐（缺配料/调料也可做）
  • almostReady = canDraw 但有缺

新鲜度系统（freshness-utils.js）

按食材名称 / 类别推算默认保质天数，入库时计算 expireAt。展示剩余天数，2天内=soon，过期=expired。过期食材不参与匹配。

菜单页首部显示临期食材提醒，推荐优先用快过期食材做菜。

1
冰箱截图 OCR 识别导入
拍一张冰箱/采购小票照片，自动识别食材并批量加入库存

用户操作流程

  1. 冰箱页新增"扫描导入"按钮，用户选择/拍照
  2. 图片上传到云存储，调用云函数执行 OCR 识别
  3. OCR 返回识别出的食材名称列表，展示确认弹窗
  4. 用户可勾选/编辑/删除误识别项，确认后批量写入冰箱库存

技术实现

核心障碍：微信小程序前端不能直接调用第三方 HTTP API，需要通过云函数作为代理。

方案 A（推荐）：调用 Claude Vision API

云函数接收图片 base64，调用 claude-haiku-4-5（成本低、速度快），prompt 为"识别图中食材，输出 JSON 数组"。

优点：识别准确，可理解中文超市小票、冰箱摆放照。
缺点：需要 Anthropic API Key。

方案 B：腾讯云 OCR

微信云开发内置腾讯云 OCR（通用印刷体识别），不需要额外账号，但识别结果是原始文字，需要再用规则/关键词提取食材名称。

优点：无需额外 API Key。
缺点：冰箱照片识别率低，需要后处理逻辑。

需要新建/修改的文件

  • cloudfunctions/ocrFridge/index.js（新建）— 云函数，调用 OCR/Vision API，返回食材列表
  • pages/fridge/fridge.js（修改）— 新增"扫描"入口、处理识别结果、批量保存

八、实现优先级建议

两个功能可以独立开发，没有依赖关系。建议顺序：

  1. 先做功能2（分享下单）的核心路径，不含通知：分享页 + 提交订单 + 厨师查看订单列表。这只需要新建几个页面和一个 CloudDB 集合，不依赖任何第三方 API。
  2. 再补全功能2的通知：申请订阅消息模板后，加上云函数推送。
  3. 然后做功能1（OCR）：先确定用哪个 AI API，再写云函数，最后连接前端 UI。

九、开发注意事项

  • 云函数必须在微信开发者工具中创建目录并上传部署，不能只在编辑器里写文件。
  • openid 获取：在小程序前端调用 wx.cloud.callFunction({ name: 'getOpenid' }) 可获取当前用户 openid（需有对应云函数，或直接用 CloudDB 查询时 _openid 字段会自动填充）。
  • 分享链接不能携带任意字符串，path 参数需 URL encode，且总长度有限制（约 1024 字节）。如果 openid 太长，可以考虑生成一个短 token 存到 CloudDB 再通过 token 查厨师信息。
  • CloudDB 权限：orders 集合建议"所有用户可写（仅限自己创建的记录），管理员/指定 openid 可读全部"。实际上只能用"所有用户"权限，厨师过滤自己的订单通过 where({ chefOpenid: '{openid}' }) 实现。
  • 当前没有使用 tabBar（底部导航），页面间用 wx.navigateTo，新页面沿用这种方式即可。
  • 本地缓存 key 已有：dishes, fridge_items, dishes_initialized, dishes_cloud_migrated, fridge_basic_guide_done。新功能新增 key 时注意不要冲突。
