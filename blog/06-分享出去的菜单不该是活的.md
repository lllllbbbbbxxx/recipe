# 分享出去的菜单,不该是"活"的

"随手做菜"有个功能:把自己会做的菜分享给朋友,朋友像点外卖一样选菜下单。第一版实现得很直觉:分享链接带上我的用户 ID,朋友打开时,实时拉取我的菜谱库。

测试的时候发现不对劲:之前发出去的链接,再打开,里面多出了我后来新加的菜。技术上这完全正确——链接本来就是一次实时查询。但就是感觉很怪。

怪在哪?是动词的语义。"分享"一份东西,在人的直觉里是快照:发给你的照片,不会因为我后来修了图而变化。当然也有活的分享——发个在线文档、甩个直播间链接,你默认它会变;但那是分享"一个入口"。这份菜单不是入口,是一份东西:这顿饭请你点,就是那一刻的这几道菜。而我把它实现成了引用——你打开的永远是"此刻的我"。数据语义和用户心智错了位,交互上怎么修补都别扭。

改法是把分享变成真正的快照:点分享时先勾选这次要给对方哪些菜,生成一份冻结的菜单,链接指向这份快照。改完之后,一堆之前没想到的东西自动顺了:有了"分享记录",可以把上次那份菜单原样再发给另一个朋友;同一个人可以按场景收到不同的菜单——给同事一份快手菜,给爸妈一份硬菜。"原样"这两个字,在实时引用的模型下根本无从谈起。

实现上有个小坑:微信的分享回调基本得当场给出链接,异步的口子有超时,等云端建快照赌不起。解法是先同步生成快照的 ID 塞进链接,快照本身丢到后台慢慢写——朋友总要几秒后才点开,来得及。还有个更隐蔽的:系统自带的转发会把上一次的快照 ID 原样带走,只好让 ID 用后即焚。

教训记下来:交互别扭的时候,先别急着改交互,往下看一层,很可能是数据语义错了。引用还是快照?用户的直觉,早就替你选好了。
