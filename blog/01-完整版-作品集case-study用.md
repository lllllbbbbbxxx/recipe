# 没有人想知道自己还剩多少克盐

> 为什么"冰箱管理"是个伪需求陷阱,以及我怎么亲手把它降级的

## 一个看起来完美的闭环

今年我用 AI 辅助开发了一个小程序,叫"随手做菜"。最初的设想朴素且诱人:把冰箱里的食材录进去,系统根据库存推荐今天做什么菜,做完自动扣减,快过期的提醒你先吃掉。

买菜 → 录入 → 推荐 → 做饭 → 扣减 → 再推荐。

一个完美的闭环。逻辑自洽,场景真实,顺手还解决了食材浪费。我当时觉得这就是产品的核心价值,库存就是护城河——毕竟,知道你冰箱里有什么的推荐,总比瞎猜的推荐准。

两个月后,我亲手把这个"核心功能"降级成了辅助输入。

## 打地鼠:补丁越打越多,越该怀疑方向

先交代一句:产品还没上线,下面所有"用户问题"都来自我自己当用户的两个月,是 n=1 的体感。但正因为连开发者本人都撑不住,才更说明问题。

问题不是一次性暴露的,更像打地鼠:按下一个,旁边又冒出一个。

**第一个补丁:录入太麻烦 → 做 OCR。** 手动一样样输入食材,我自己录了两次就不想录第三次——开发者本人都坚持不了的事,很难指望用户坚持。于是接了 OCR,拍购物订单截图自动识别。听起来很 AI,很优雅。

**第二个补丁:OCR 出来全是垃圾 → 做噪声过滤。** 真实的购物截图识别出来是这样的:`21:46`、`¥1.59`、`购物车▼`、`已售罄`、`天天红包`……十四行里只有三行是食材。于是又写了一整套后处理:过滤价格、店名、广告词,提取"菜名+重量",再过一遍食材词库。

**第三个补丁:数量没法扣 → 重做数量模型。** 库存里存的是"3个""200g""适量"这种自由文本,想做自动扣减时发现根本没法算。于是把数量拆成"数字+单位"两个字段,老数据还得做兼容解析。

**第四个补丁:库存只增不减 → 做消耗闭环。** 我自己试用时,不做扣减的库存不出几天就和真实冰箱对不上了,失真的库存让推荐变成负资产。于是做了"做完菜按量扣减":计数类按菜谱用量默认扣,重量类选"用了一半 / 用了 1/4 / 自定义",适量不扣。

**第五个补丁:盐和酱油不该被管理 → 引入常备概念。** 没有人想知道自己还剩多少克盐。于是又加了"常备食材"标记,常备的不参与数量管理……

单看每一个补丁,都是合理的产品迭代,做完都有成就感。但连起来看是另一个故事:**每个补丁都在回答"怎么让用户维护库存",没有一个补丁回答"用户为什么要维护库存"。**

## 转折点是一份自己写的 PRD

转折不是来自用户反馈,而是来自写 PRD。

写到风险评估那一节,我写下了这样两句:

> 整个产品成立的前提,是"用户愿意持续维护库存"。如果 OCR 入库 + 自动推断消耗都无法解决这个问题,需要考虑调整为"不需要库存数据的轻量版"路线。

旁边风险表里,这一条的评级是:可能性,高;影响,致命。

写完盯着这行字看了很久。刺眼的不是"最大风险是核心前提"——任何早期产品的最大风险都是核心假设,这是常态。刺眼的是:**我为这个致命前提打了五个补丁,却从没为它设计过任何一个验证手段。** 补丁在替一个未经验证的直觉付利息,而我连"它到底成不成立"的测量方法都没有。

## 算一笔账:口头需求 vs 行为需求

先把词说清楚。我说的"伪需求"不是"需求不存在",而是特指一类:**用户口头上认同、但不愿意为它持续付出行为的需求。** "你想不想清楚知道冰箱里有什么?"——人人都说想。"你上一次主动记录冰箱是什么时候?"——沉默。需求存在于口头认同层面,不存在于行为层面。

用账本看得更清楚。

维护库存的**成本**:每次买菜要拍照、确认、勾选,每次做饭要确认扣减,偶尔清理过期数据。长期、高频、没有即时反馈。

获得的**收益**:推荐更准一点,临期提醒偶尔有用。低频、边际、可被替代——不知道库存,推荐照样能做,只是从"你能做这道菜"退化成"这道菜你多半能做"。

成本是持续的税,收益是偶尔的糖,这笔账平不了。而且没有人会因为"想管理库存"打开一个 app——用户打开它的时刻,是晚上六点站在厨房门口不知道吃什么的时刻。**把"你希望用户顺便做的事"当成产品核心,就是掉进伪需求陷阱的标准姿势。**

我没做过系统的竞品调研,但搜过一圈同类产品,印象最深的是差评几乎指向同一件事:录入坚持不下来。我怀疑这个品类萧条的主因不是需求不存在,而是维护成本大于收益。记账工具是个有意思的参照:口头上想记账的人极多,坚持手记的人极少;但手动记账也有活得很好的——比如 YNAB,它把"手动录入"本身做成了强迫觉察的仪式,录入这个动作自己就产生即时反馈。这恰好说明出路只有两条:**要么把维护自动化到零,要么让维护行为本身即时可感。** 冰箱管理两条都不占:自动化有下限(下面细说),而"录完一袋菜"没有任何爽感。

## 降级,而不是删除

想清楚之后,重新定位反而简单。

主定位从"冰箱库存管理"换成"做饭决策助手"——用户的真实高频问题是"今天吃什么",不是"我有什么"。首页围绕决策:抽卡、推荐、今日菜单;库存入口下沉。

库存呢?降级为推荐的辅助输入。有库存数据,推荐更贴合、能优先消耗临期食材;没有,产品照样能用。它从成立前提变成了锦上添花。

设计原则跟着换:允许模糊、近似、偶尔更新。不追求仓库级精确,只求"大概知道有什么"。常备食材不管数量,数量随意填,"适量"就不扣。产品不再要求用户当仓库管理员。

AI 的位置也清楚了:不是炫技,是替用户支付录入成本。但这里有个我之前没想透的关键——**OCR 和自动化只能把成本从"记账"压到"点头确认",压不到零。** 拍照、勾选识别结果、确认扣减,这些交互成本消不掉。这正是库存只配做辅助输入、做不了核心的根本原因:一个成本下限大于零、收益又是边际的功能,永远算不平那笔账。

注意,是降级,不是删除。理论上库存数据应该能让推荐更准——前面的补丁没有白打,它们把维护成本压到了我目前能做到的最低。错的从来不是功能本身,是它在产品里的位置。

## 诚实的部分:新判断也还没被验证

必须坦白:"降级后的定位更好",目前也只是一个假设。产品即将开始第一轮 10-15 名体验者的验证,还没有任何真实数据。

旧判断和新判断,本质上都是直觉。区别只有一个:这次我提前写死了怎么量、量到多差就撤。

- 埋点里有一组对比:维护库存的用户 vs 不维护的用户,推荐接受率差多少。严格说这是相关性对比,不是实验——愿意维护库存的人本来就更投入,所以我会按活跃度分层之后再看,差距依然明显才算数。
- 止损线是量化的:库存维护率低于 30%,或做完菜扣减率低于 20%,就启动"无库存轻量版",彻底砍掉库存,只留抽卡决策和分享点菜——这条退路,正是 PRD 风险评估那两句话里写好的。

上一次,我在没有验证的直觉上盖了两个月的楼。这一次,至少地基上装了传感器。

## 三个自查,识别你产品里的同类陷阱

**一、打开动机测试。** 一个需要用户持续手动投入、又不构成打开动机的功能,不能当核心定位(注意限定:全自动的能力不适用这条)。提问方式也有讲究——别问"你觉得 X 有用吗",人人都说有用;要问"你上一次主动做 X 是什么时候"。

**二、维护成本账。** 这个功能要求用户长期、高频、无即时反馈地付出吗?是的话,默认按伪需求处理——除非你能把它做到全自动、无需用户确认,或者让维护行为本身即时可感。"接近零成本"不算豁免:从记账到点头确认,省的是力气,不是动机。

**三、看补丁的性质,而非数量。** 打降成本补丁之前先回答:用户有做这件事的动机、只是被摩擦挡住了吗?如果诚实的答案是"我希望他有",停下来。补丁数量本身不说明问题(电商结账流程优化几十轮都是对的),但它是个信号:当你发现连续几个补丁都在降低同一件事的成本,值得把这个问题拿出来问一遍。

最后,把现在贴在项目文档最上面的产品原则抄在这里——它就是这次降级的全部结论:

> 不要让用户管理厨房,让产品帮用户决定今天吃什么。

用户不缺工具,缺的是有人替他把那个每天重复的小决定做掉。
