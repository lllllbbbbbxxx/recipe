# 识别只是入口,后处理才是产品

给冰箱做"拍购物截图自动入库"时,我在腾讯云 OCR 和大模型 Vision 之间选了前者:有免费额度、快、输出稳定。事后看,这个选择没那么重要——真正的战场在识别之后。

第一次真机测试,一张买菜订单截图,OCR 忠实地吐出了十四行:21:46、69、购物车▼、管理、超出配送范围、已售罄、¥1.59、有机迷你杏鲍菇200g、王小菜(市北市场门店)……十四行里,食材三行。

然后是三轮迭代。第一轮很天真:按文本长度过滤,一到十二个字的保留——结果"管理"这种两个字的按钮完美通过。第二轮打黑名单:正则去掉价格、时间、纯符号,再配一个噪声词库(购物车、已售罄、满减、结算……)——真机一测,"大口""天天红包""消息"还是漏了进来。黑名单永远列不完,你不知道下一张截图会带来什么新按钮。第三轮换思路,改白名单:只保留命中食材词库的条目,顺带把"杏鲍菇200g"拆成名称加重量,去掉"(介意肥肉的勿点)"这种括号备注和"新鲜香菜约"拆剩的那个"约"。回放那几张截图,进来一屏,出去的正好是食材——当然,前两轮我也是这么以为的,这次至少换对了方向。

白名单有它对称的代价:词库没收录的食材会被误杀。这个坑我先记着——至少误杀比放"天天红包"进冰箱,更容易被用户发现和原谅。

回头看,OCR 这一步没给我带来任何差异化——在这个场景里它只需要"能用",而现在主流 API 基本都能用。demo 和产品之间隔着的,是那个词库、那几条正则,和"黑名单列不完就换白名单"的判断。

模型谁都租得起,那个词库和那几条正则才是你自己的。用户永远看不见它们——但漏进冰箱的"天天红包",一眼就看得见。
